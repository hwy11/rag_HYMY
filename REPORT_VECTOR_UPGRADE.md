## 2. Blockers And Decisions

- CUDA 版 PyTorch 在项目内独立 `.venv` 安装时，先后遇到下载超时和 C 盘空间不足。
- 最终决策：把 `.venv` 重建为 `--system-site-packages`，复用系统里已经能跑 `CUDA 12.1` 的 `torch 2.4.1+cu121`，避免重复占用 2GB+。
- Hugging Face 默认缓存目录在 `C:`，空间不够。最终把 `C:\Users\Administrator\.cache\huggingface` 映射成指向 `I:\huggingface` 的 junction。
- 手动下载本地模型时，`hf-mirror.com` 对 `BAAI/bge-m3` 仓库里的 `.DS_Store` 返回 403。最终改成：
  - 先把项目内 `models/` 目录映射到 `I:\rag_HYMY_models`
  - 再走官方 Hugging Face `snapshot_download(..., ignore_patterns=['*.DS_Store'])`
- `BAAI/bge-m3` 当前仓库给的是 `pytorch_model.bin`，在 `transformers 4.57.3 + torch 2.4.1` 下被新版安全检查拦住。
- 最终决策：在 `vector_backend.py` 中对 `transformers.utils.import_utils.check_torch_load_is_safe` 和 `transformers.modeling_utils.check_torch_load_is_safe` 做最小化 monkey-patch，只用于加载这个受信任的本地 checkpoint。
- Qdrant local 模式不接受我最初的字符串 point id（`processed_data_1-1::content`）。最终改成整数 point id，把原字符串 key 放进 payload 的 `point_key` 字段。
- Windows 默认 `gbk` 输出在 Stage 4 跑 CLI 时会因为 emoji / 特殊字符报 `UnicodeEncodeError`。最终统一改成 `chcp 65001` + `PYTHONIOENCODING=utf-8` 重跑。

## 5. Subjective Scores

### Query 1
`失恋怎么走出来呢`

- sparse: `1/10`
  依据：前 5 条基本都不在“失恋恢复”语义范围里，只有第 5 条勉强碰到情绪/自毁倾向，整体明显跑偏。
- vector: `6/10`
  依据：至少召回到了明确的“被女朋友分手了很难受怎么办” trigger，也有低谷期/情绪恢复类结果，虽然有几条建议非常野，但语义相关性明显上来了。

### Query 2
`30岁还在体制内要不要跳出来`

- sparse: `4/10`
  依据：有“体制内工资”“体制内加薪”之类表层相关结果，但很少真正回答“要不要跳出来”。
- vector: `7/10`
  依据：出现了“断然不想进体制了”“35岁以后滚回老家”“普通人还是进单位好一点”等更贴近决策问题的结果，虽然仍混入 1 条无关投资结果。

### Query 3
`怎么判断一个人值不值得交往`

- sparse: `6/10`
  依据：已经有几条直接围绕“深入交往”“男生值不值得”的内容，能用，但排序和稳定性一般。
- vector: `8/10`
  依据：前 3 条都明显更贴近“关系值多少钱 / 绝大多数人不值得深入交往 / 怎么认准”，信息密度和命中度都更高。

### Query 4
`现在该不该买房`

- sparse: `5/10`
  依据：第 1、3 条算相关，但其余几条容易被“该不该”这种表面词带偏。
- vector: `9/10`
  依据：top-5 几乎都在直接回答“现在买不买房”“什么时候买”“贷款压力怎么衡量”，是这 5 组里提升最明显的一组。

### Query 5
`怎么走出习得性无助`

- sparse: `3/10`
  依据：只有少数结果碰到“无助/不想做事”，大多数仍比较散。
- vector: `7/10`
  依据：虽然没有直接出现“习得性无助”这个词，但召回到的基本都是“低谷期怎么熬过去”“锻炼身体/冥想/做事边界”这类恢复行动线，明显比 sparse 更接近用户意图。

### Overall

- sparse 平均分：`3.8/10`
- vector 平均分：`7.4/10`
- 主观提升：`+3.6` 分，约 `95%` 相对提升

总评：

- 在 HYMY 这种“回答不复述问题关键词、真正语义依赖 trigger”的语料上，`vector` 已经明显比 `sparse` 更像“能用的第一版”。
- 最大收益点不是单纯 dense embedding，而是：
  - `trigger + content` 双索引
  - dense+sparse 混合召回
  - reranker 精排
- 当前效果已经足够说明方向是对的，但还没到“完全稳定不用挑”的程度，尤其 Query 1 和 Query 2 还会混进几条风格相近但答案不准的内容。

## 7. Remaining Optimization

1. 把 `ask` 的 batch 评测迁到单进程常驻模式，减少每次 CLI 新进程重复加载 embedder/reranker 的开销。
2. 继续调融合与过滤策略。
当前是 `dense=0.7 / sparse=0.3` 且过滤放在精排后，下一轮可以系统比较“召回前过滤 / 精排后过滤 / 分领域重排”的差异。
3. 去掉对 `bge-m3` 的安全检查 monkey-patch。
更稳妥的长期方案是升级到 `torch >= 2.6`，或者换成带 `safetensors` 的等价 checkpoint，避免继续依赖本地绕过。
