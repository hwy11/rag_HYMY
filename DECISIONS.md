项目当前的关键决策是偏“先把闭环做实，再追求更强模型能力”，所以有些选择是有意保守的。

向量库这件事，当前版本**没有上 Qdrant，也没有上 Chroma**。我先用了本地 JSON 索引加稀疏检索，原因不是它更先进，而是它零依赖、零服务进程、最容易把清洗、打标、蒸馏、提问包这条链路一次跑通。等真实数据规模上来以后，再把检索层替换成 Qdrant 或混合检索，迁移成本可控，因为上游数据结构已经稳定了。

chunk 这件事，当前版本**不做二次切片**，一条回答就是一个基本检索单元。原因是你给的原始材料天然就是短到中等长度的“问题-回答”片段，现在最怕的是切碎后丢掉语义完整性，尤其是价值观、方法论类表达。等真实数据里出现超长回答，再补基于语义边界的切片策略。

打标字段目前选 `domains`、`type`、`time_sensitivity`、`confidence`、`key_concepts`、`one_line_summary`，因为这几个字段分别服务于三个后续动作：检索召回、蒸馏筛选、提问包校准。`type` 和 `time_sensitivity` 是最关键的两个字段，它们直接决定什么能进入思维 DNA，什么只能作为带时效的参考材料。

persona 设计成 `lite / standard / full` 三档，不是为了好看，而是为了控制网页大模型上下文预算。Lite 适合日常快问，Standard 适合正常决策，Full 适合深度长问。这个分层比“永远把全部 persona 塞进去”更稳。

成本估算先做成本地 dry-run，而没有接真实 API，是因为当前阶段最需要先知道数据量会把你带到什么量级，而不是立刻把钱烧掉。等你给真实 1 万条语料，再接你指定的模型供应商更合理。

## 检索层升级决策

检索层这次升级到 `BGE-M3 + Qdrant + reranker-v2-m3`，核心原因不是“追新”，而是原来的稀疏 token 检索在 HYMY 语料上天然吃亏。很多回答根本不会重复问题里的关键词，真正决定语义的是原提问 trigger、隐含语境和博主的跳跃式表达，所以只靠关键词命中，top-1 经常会偏到完全无关的语录。

Embedding 选 `BGE-M3` 而不是 `Qwen3 embedding`，是因为这次任务最需要的是一个能稳定输出 dense + sparse 的单模型方案。`BGE-M3` 已经把混合检索路径跑成熟了，和 `FlagEmbedding` / `Qdrant` 的配合也更直接，工程不确定性更低。现在先把“混合召回 + 精排”这条链路跑稳，比再花一轮时间对比新 embedding 家族更重要。

向量库选 `Qdrant` 而不是 `Chroma`，主要是因为这次明确需要 named vectors、sparse vectors、本地文件模式和后续混合检索扩展。`Qdrant` 在这些能力上更贴近目标架构，而且可以不额外起服务，直接落到本地目录，保留了当前项目“单机、可迁移、低运维”的优点。

混合召回权重先定成 `dense=0.7 / sparse=0.3`，不是说这个比例永远最优，而是把语义召回放主导、关键词召回做纠偏。HYMY 语料里，纯 sparse 容易被表层词带偏；纯 dense 又可能把“情绪像、措辞像”的内容混进来。先让 dense 占七成，把 sparse 当作召回校准，是当前这批语料更稳妥的默认值。

Reranker 不是“锦上添花”，而是这套方案能不能真正超过 sparse 的关键。召回阶段负责“别漏掉”，精排阶段负责“把最像答案的那条放上来”。尤其在同一个 query 会召回多条都“有点像”的情况下，不加 reranker，最终前几条结果的观感波动会非常大。
